# Завдання: Сторінка "Повна програма курсу" на qastudy.online

## Контекст

Окремо від лендінгу курсу "Performance Testing з k6" (короткий опис +
акордеон з назвами занять) потрібна повна сторінка програми — з повним
текстом кожного заняття (теорія, практика, ДЗ, результат) та фінальним
проєктом. Контент готовий у файлі `k6-full-program-content.md`
(переданий разом з цим завданням).

---

## КРОК 0 — Дослідження (якщо лендінг курсу ще не створено — спочатку
виконати завдання з файлу `k6-course-landing-instructions.md`, ця сторінка
є його продовженням)

1. Знайти, де лежить файл лендінгу курсу (наприклад
   `content/courses/k6-performance-testing.md` чи
   `content/courses/k6-performance-testing/index.md` — залежно від того,
   яку структуру обрали на попередньому кроці)
2. Перевірити, як на сайті влаштовані "довгі" текстові сторінки (документація,
   статті блогу) — який layout вони використовують, чи є типографічні стилі
   для `<h2>/<h3>`, таблиць, `<blockquote>`, які варто переюзати тут
   (повна програма — це насамперед текстовий контент, не картки/гріди)

---

## КРОК 1 — Розміщення файлу та front matter

Розмістити `k6-full-program-content.md` як:

- `content/courses/k6-performance-testing/program.md` — якщо лендінг
  зроблено як `content/courses/k6-performance-testing/index.md` (page bundle)
- або `content/courses/k6-performance-testing-program.md` — якщо лендінг
  це окремий файл `content/courses/k6-performance-testing.md`

Підлаштувати шлях так, щоб фінальний URL був логічним продовженням URL
курсу, наприклад: `/courses/k6-performance-testing/program/`.

**Front matter:**

```yaml
---
title: "Повна програма курсу: Performance Testing з k6"
description: "Детальна програма курсу Performance Testing з k6 для DevOps та QA: 15 занять, 4 блоки — від НФР до Chaos Engineering з xk6-disruptor."
registration_url: "PASTE_GOOGLE_FORM_LINK_HERE"
---
```

`registration_url` — та сама змінна, що й на лендінгу. Якщо на лендінгу
вже визначили `registration_url` у `Site.Params` (глобально для всього
сайту/курсу) — на цій сторінці front matter для `registration_url` не
потрібен, прибрати рядок і переконатись, що шаблон бере значення з
`Site.Params.registration_url`. Якщо ж на лендінгу це page-level параметр —
скопіювати те саме значення сюди (або, що краще, винести в один спільний
`data`-файл/`Site.Params`, щоб не дублювати посилання у двох місцях).

---

## КРОК 2 — Зв'язок з лендінгом

1. На лендінгу курсу (секція 2.6 "Повна програма курсу" / заключний CTA
   з `k6-course-landing-instructions.md`) додати посилання на цю нову
   сторінку: `[Дивитись повну програму курсу →](/courses/k6-performance-testing/program/)`
   (підставити реальний фінальний URL після Кроку 1)
2. На новій сторінці програми вгорі та внизу вже є посилання
   `[← Назад до опису курсу](/courses/k6-performance-testing/)` —
   перевірити, що цей шлях відповідає реальному URL лендінгу, і
   виправити за потреби
3. Якщо на сайті є головна навігація/меню курсів — сторінку програми
   **не додавати** в головне меню (вона є дочірньою сторінкою курсу,
   доступною через лендінг), щоб не дублювати пункти меню

---

## КРОК 3 — CTA-кнопки

На сторінці програми вже є дві кнопки `Залишити заявку`
(`class="cta-button"`, вгорі та внизу) — переконатись, що:

- використовується той самий клас/стиль `.cta-button`, що визначений
  для лендінгу (не створювати новий стиль кнопки)
- посилання веде на `registration_url` (через `{{ .Params.registration_url }}`
  або `{{ .Site.Params.registration_url }}` — залежно від рішення з Кроку 1)
- кнопка відкривається в новій вкладці (`target="_blank" rel="noopener"`)

---

## КРОК 4 — Типографіка довгого контенту

Сторінка складається переважно з `##`/`###` заголовків, абзаців,
`**bold**`-лейблів ("Практика на занятті:", "ДЗ:", "Результат після
заняття:") та однієї таблиці (матриця охоплення тестами). Якщо в темі
сайту вже є стилі для `<h2>`, `<h3>`, `<table>`, `<blockquote>` —
переконатись, що вони застосовуються коректно (відступи, читабельна
ширина рядка, таблиця не "розповзається" на мобільних — за потреби
обгорнути `<table>` в контейнер з `overflow-x: auto`).

Якщо в темі немає стилю для `<blockquote>` (використовується для цитат
JTBD у розділі "Цільова аудиторія") — додати простий стиль:

```css
blockquote {
  border-left: 3px solid var(--color-primary, #2563eb);
  padding-left: 1rem;
  margin: 1rem 0;
  color: var(--color-text-muted, #6b7280);
  font-style: italic;
}
```

## КРОК 5 — Перевірка

1. `hugo server -D` — переконатись, що сторінка будується за очікуваним
   URL і весь контент (15 занять + фінальний проєкт) відображається
2. Перевірити якірні посилання в розділі "Зміст" — клік по кожному пункту
   має скролити до відповідного блоку/заняття
3. Перевірити мобільну ширину (~375px) — таблиця матриці охоплення тестами
   не повинна ламати верстку (горизонтальний скрол прийнятний)
4. Перевірити обидва переходи: лендінг → програма (через нове посилання
   з Кроку 2) і програма → лендінг (через "← Назад до опису курсу")
5. Перевірити обидві CTA-кнопки на сторінці програми — ведуть на
   `registration_url`
6. `hugo --minify` — білд завершується без помилок

## ЧОГО НЕ РОБИТИ

- Не дублювати повний текст програми ще й на лендінгу — лендінг має лише
  короткий акордеон з назвами занять і посилання на цю сторінку
- Не хардкодити `registration_url` повторно як окреме значення, якщо вже
  є спільний `Site.Params.registration_url` — уникати дублювання джерела
  істини для посилання на форму
- Не додавати сторінку програми в головну навігацію сайту
- Не змінювати порядок або текст занять із наданого файлу
  `k6-full-program-content.md` — якщо щось не влазить у дизайн (наприклад,
  таблиця), вирішувати через CSS/обгортки, а не через скорочення контенту
- Не вигадувати додаткові секції (відгуки, FAQ тощо) — це поза межами
  завдання
