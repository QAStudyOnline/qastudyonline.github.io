# Завдання: Лендінг курсу "Performance Testing з k6" на qastudy.online

## Контекст

Сайт qastudy.online збудований на Hugo та хоститься на GitHub Pages.
Потрібно додати новий лендінг для курсу "Performance Testing з k6 (для
DevOps та QA)" — 15 занять, 4 блоки, рівень Middle–Senior.

## КРОК 0 — Дослідження репозиторію (ОБОВ'ЯЗКОВО ПЕРЕД БУДЬ-ЯКИМИ ЗМІНАМИ)

Перш ніж створювати будь-які файли, дослідити структуру репозиторію:

1. Подивитись `hugo.toml` / `config.toml` / `config.yaml` (де лежить конфіг сайту)
2. Подивитись структуру `content/` — як організовані існуючі сторінки/лендінги
   (чи є вже подібні лендінги курсів — взяти один за зразок структури front matter)
3. Подивитись `layouts/` — яка тема використовується, чи є кастомні layouts/partials
   для курсів, чи є готовий accordion/FAQ-компонент, чи є готова CTA-кнопка/компонент
4. Подивитись `assets/` або `static/` — як підключаються кастомні CSS/SCSS
   (Tailwind? власні SCSS-файли? CSS-змінні теми?)
5. Перевірити `tailwind.config.js` або `_variables.scss` / `:root` зі CSS-змінними —
   які кольори, типографіка, spacing-токени вже визначені, щоб новий лендінг
   виглядав консистентно з рештою сайту, а не як окремий шматок

Якщо знайдено існуючий лендінг курсу — використати його файл як шаблон
структури (front matter, секції, іменування файлів), а не вигадувати нову
структуру з нуля.

кнопка все одно рендериться (не ламає білд), просто веде на placeholder —
власник сайту замінить значення після деплою.

### 2.2 Для кого цей курс (Audience)

Дві картки (по одній на сегмент аудиторії):

**Картка 1 — QA-інженери (middle/senior)**
- Хук (JTBD): *"Я вмію тестувати функціонал, але коли мене питають 'а як
  воно поводиться під навантаженням' — я губляюсь і не знаю, з чого почати"*
- Що отримаєш:
  - Формулювати НФР і переводити бізнес-вимоги в цілі навантаження
  - Писати та підтримувати k6-тести (протокольні та браузерні)
  - Читати метрики й писати запити InfluxQL/PromQL, будувати дашборди Grafana
  - Знаходити регресії перформенсу до релізу

**Картка 2 — DevOps-інженери (middle/senior)**
- Хук (JTBD): *"У нас є CI/CD і моніторинг, але перформенс тести хтось
  запускає вручну раз на квартал — хочу, щоб це працювало автоматично і
  давало зрозумілий результат"*
- Що отримаєш:
  - Observability-стек для перформенс тестування (k6 → storage → Grafana)
  - k6 у CI/CD як gate перед релізом
  - Розподілені тести в Kubernetes через k6-operator
  - AI-аналіз результатів та chaos-експерименти з xk6-disruptor

### 2.3 Чого навчишся (What you'll learn)

5–7 карток/іконок з ключовими компетенціями (НЕ список усіх 15 занять):

1. Формулювання НФР і побудова процесу перформенс тестування на проєкті
2. Написання якісних навантажувальних тестів у k6 (включно з AI-генерацією)
3. Глибокий аналіз метрик через InfluxDB (InfluxQL) та Prometheus (PromQL)
4. Побудова дашбордів у Grafana, включно з порівнянням run-to-run
5. UI Performance Testing через k6 browser та Web Vitals
6. Інтеграція k6 в CI/CD та розподілені тести в Kubernetes (k6-operator)
7. AI-агент для аналізу результатів + Chaos Testing з xk6-disruptor

### 2.5 Фінальний проєкт (checklist)

Рендерити як простий чеклист (`<ul>` з іконками-галочками або стилізовані
`<li>`), без чекбоксів-інпутів:

- НФР та процес — документ з НФР для 3–5 сценаріїв + опис процесу перформенс тестування
- Протокольні k6-тести — мінімум 3 сценарії з checks і thresholds (частково AI-генеровані)
- Browser-тест(и) — k6 browser-тест з вимірюванням Web Vitals
- Observability-стек — custom load-профіль, tags/groups, метрики в InfluxDB та Prometheus
- Дашборди — стандартний + кастомний + порівняльний (run-to-run)
- CI/CD пайплайн — k6-тести з thresholds як gate
- Kubernetes-запуск — TestRun через k6-operator
- AI-аналіз — звіт від AI-агента (Grafana MSP)
- Chaos-експеримент — load-тест + xk6-disruptor, звіт про стійкість системи

## КРОК 4 — Перевірка

1. `hugo server -D` — переконатись, що сторінка курсу будується і доступна
   за очікуваним URL
2. Перевірити мобільну ширину (~375px) — картки audience/skills та accordion
   мають коректно перевлаштовуватись
3. Перевірити, що CTA-кнопки (hero + closing) ведуть на значення
   `registration_url` з front matter
4. Перевірити, що при відкритті сторінки перший `<details>` (Блок 1)
   відкритий, решта закриті, і що клік по `<summary>` розгортає/згортає
   блок без JS-помилок у консолі
5. Зібрати продакшн-білд: `hugo --minify` — переконатись, що білд завершується
   без помилок
6. Перевірити, чи нова сторінка з'явилась у навігації/списку курсів сайту
   (якщо на сайті є меню/каталог курсів — додати туди посилання за тим же
   шаблоном, що й інші курси)

- Не хардкодити URL гугл-форми прямо в HTML/шаблоні — тільки через

  семантичний `<details>`/`<summary>`

- Не вигадувати посилання на повну програму курсу, якщо такого документа/
  сторінки немає в репозиторії — просто не додавати цей елемент
- Не змінювати глобальну тему/кольорову схему сайту — новий лендінг має
